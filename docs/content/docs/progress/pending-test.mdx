## Canvas 与 Workflow 双系统

- Canvas 已移除节点引用连线可视化，保留原有 `@` 引用数据和生成传参，并继续使用元素历史与 SVG/Konva 渲染路径。
- Canvas 与 Workflow 复用同一套顶部菜单状态与命令，一键切换工作区，并分别保存当前画布或当前 Workflow 项目。
- 顶部菜单统一承载 Flovart 标识、项目标题、API 能力状态、语言、主题和设置；Canvas 原底部状态与重复菜单已移除。
- Canvas 工具栏移动到底部并支持紧凑换行；裁剪确认、取消和形状/画笔弹层保持可点击，不再被旧底部菜单或滚动容器遮挡。
- Canvas 与 Workflow 左右抽屉按各自工作区定位和缩放，不再使用叠加的固定顶部偏移。
- Canvas 关闭左侧抽屉后，双击聚焦和适配视图不再错误预留侧栏宽度；开始拖拽或滚轮操作会立即停止尚未结束的视口动画。
- Canvas 元素拖拽在松手前复制最终偏移再提交，避免异步状态更新读取到已清空的偏移而回跳原位。
- Workflow 支持独立多项目、图片/文本/视频/音频/配置节点、拖拽缩放、框选、多选对齐与层级、复制粘贴、撤销重做、连线、缩略图和背景切换。
- 双击空白处可新建节点；从节点连线拖到空白处可选择节点类型并一次完成创建、连接和选中。
- 图片、视频和音频可直接拖入画布、从工具栏导入、粘贴或替换；拖入支持浏览器空 MIME 文件的扩展名识别，媒体使用本地持久化并在缺失时显示可恢复状态。
- 点击图片内容或视频播放控件会先选中节点并显示 Flovart 自己的 PromptBar 和 ElementToolbar；PromptBar 支持 `@` 引用、模型、批量、状态、停止和重试。
- Workflow 可导入/导出包含本地媒体的严格 JSON，导入会重建媒体存储；删除项目需要二次确认。
- 共享素材支持搜索、图片/视频筛选和预览，并提供内置 Prompt Library。
- Workflow 右侧面板默认展开并记住开关和宽度，复用 Canvas 的紧凑面板外壳；嵌入式 Agent 的模式栏和导航栏改为透明背景无边框，与抽屉 Tab 栏视觉统一，不再出现双层 header 叠摞。
- Workflow 节点已移除顶部标题栏（图片/视频等标签边框），节点为纯媒体/纯内容；错误提示移至右上角小角标，拖动改为整节点可拖。
- Workflow 底部工具栏已移除上传图片/视频/音频按钮，改用 Canvas 的 `isl-icon-btn` 扁平风格，保留选择/平移/文本/配置/共享素材/撤销/重做/适应/网格/Agent 入口。
- Workflow 的生成历史与素材库支持搜索、图片/视频和素材分类筛选、点击添加、拖到画布落点、下载、反推 Prompt、重命名和删除；点击或拖入的本地媒体继续使用现有 Blob/IndexedDB 持久化，不把 base64 写进项目 JSON。
- Workflow 右侧历史/素材、画布拖入和反推 Prompt 会先解析 `idb:board:*`、`idb-video:*`、`cold-media:*` 等本地媒体引用，不再把这些 IndexedDB 协议直接交给浏览器图片/视频标签。

## Workflow 生成

- 配置节点复用 Canvas 当前 API Key、模型选择和 Provider 能力元数据。
- 上游文本和 `@` 文档会合并到提示词，上游图片、视频和音频只在当前模型支持时作为参考输入。
- 文本、图片和视频生成支持请求身份、停止、过期结果隔离、错误、重试、批量和进度状态。
- 生成成功后在右侧创建结果节点并自动连线，不覆盖来源节点；图片和视频结果写入现有生成历史。
- 图片节点提供裁剪、滤镜、高清放大、移除背景、扩图、蒙版修改和图层拆分；并发工具使用单事务，撤销历史不会串到其他节点。
- 音频媒体节点可上传、播放、引用和编辑音色/格式/语速等配置；音频生成仍明确标记为 Provider 未支持。

## Workflow 节点动效

- Workflow 节点入场/退场使用 Framer Motion spring 物理（scale 0.85→1，stiffness 380 damping 24），悬停时轻微放大 1.02；生成中状态启用呼吸动效（scale `[1, 1.015, 1]` 1.8s 循环）、脉冲遮罩和扫光条（`.workflow-node__shimmer`，Framer Motion `animate.x` -120%→120% 1.6s 循环）。
- 生成中节点禁用 `whileHover` 避免与呼吸动效冲突；选中态边框/阴影用 `cubic-bezier(.34,1.56,.64,1)` overshoot 过渡。
- Workflow 连线使用 `motion.path` + `pathLength` spring 入场（stiffness 120 damping 18）；选中连线添加 SVG 原生 `<animateMotion>` 数据流粒子和 CSS `@keyframes workflow-connection-flow` dash 流动。
- AGENTS.md 已更新：Canvas 保留生成中动画禁令，Workflow 放开限制允许弹性呼吸/脉冲/扫光等动效，动效统一使用 Framer Motion spring 物理，不手写 CSS keyframes 模拟弹性。

## Workflow 批量结果堆叠与导出

- 批量生成（count > 1）的结果节点自动分配同一 `batchId` 和 `batchIndex`；折叠态下同批次节点全部隐藏，显示堆叠卡片（两层背景偏移 + 头图预览 + 数量徽章），点击展开/折叠。
- 堆叠卡片使用 Framer Motion spring 入场/退场，`whileHover` 轻微放大；展开时各节点通过 `AnimatePresence` spring 入场。
- 折叠态下连向隐藏节点的连线同步隐藏；展开后恢复。
- 批量导出 ZIP 文件名格式改为 `项目名_节点标题_序号.ext`（通过 `exportMediaArchive` 新增 `projectName` 参数）；未传 `projectName` 时回退原 `序号_名称.ext` 格式，向后兼容。

## Provider 与模型偏好

- RunningHub 已作为原生标准模型 Provider 接入设置页、模型拉取、图片生成和视频生成网关；模型列表内置 7 个经实跑验证的标准模型端点（含 Flash 快速图生图/文生图、V3.1 Fast 视频等），page-api 拉取的模型作为补充，仍可手动补充模型 ID。
- RunningHub API Key 验证会使用标准模型任务查询端点确认鉴权，并在保存或后台刷新时写入可选模型；模型生成返回的 `results[].url` 会转换成现有图片/视频结果格式。
- RunningHub WebApp API（`/task/openapi/*`）已移除浏览器禁用的 `Host` 请求头，文件上传、AI App 提交和输出查询在浏览器环境可正常调用。
- RunningHub 标准模型调用会按详情页字段自动映射 `imageUrls`、`firstFrameUrl`、`lastFrameUrl`、`aspectRatio`、`ratio`、`videoUrls` 和 `audioUrls`；本地 data URL 参考媒体会先上传为临时公网 URL。
- RunningHub 标准模型提交、查询和上传会透出平台返回的 `errorCode` / `errorMessage`，并附带不含密钥的 `baseUrl`、最终提交 URL、模型 ID 与 payload 字段摘要；上传未返回公网 URL、模型 ID 仍是详情页/搜索页 URL，或 payload 中仍有本地引用时会在本地中断，不再退化成 `No taskId returned`。

## Phase 2.1 — Script 节点 + 工作流编排基础

### 类型与数据结构
- **新增 Script 节点类型**：`WorkflowNodeType` 新增 `'script'`，作为 Workflow 第 6 种节点类型。Script 节点定位为"剧本拆解 + 分镜编排 + 批量生成"容器，Canvas 纯视觉不背复杂状态机。
- **新增类型定义**（`types.ts`）：`ScriptAsset{id,kind:'character'|'scene'|'prop',name,description?,settingImageNodeId?}`、`ScriptShot{id,index,emotion?,action?,dialogue?,sfx?,scene?,promptOverride?,colorTag?,imageNodeId?,videoNodeId?,status?}`、`ScriptBreakdown{assets,shots,sourceText?,referenceVideoNodeId?,modelId?}`、`WorkflowBatchGroupSource='auto'|'manual'`。
- **WorkflowNode 扩展**：新增 `batchGroupSource?: WorkflowBatchGroupSource` 字段，区分手动打组（Ctrl+G）和自动打组（批量生成 count>1）。
- **WorkflowNodeMetadata 扩展**：新增 `referenceNodeIds?: string[]`（显式声明上游引用节点，覆盖连线推断）、`scriptBreakdown?: ScriptBreakdown`（Script 节点的分镜数据）。
- **新增 3 种 WorkflowOp**：`group_nodes{ids,batchId,source}`、`ungroup_nodes{ids}`、`execute_group{nodeIds}`。

### 常量与节点规格
- **Script 节点默认尺寸**：480×560，`metadata.scriptBreakdown` 默认 `{ assets: [], shots: [] }`。

### 操作与算法（ops.ts）
- **`topoSort(nodes, connections, nodeIds)`**：Kahn BFS 拓扑排序，用于整组执行时确定节点运行顺序。含 cycle fallback（检测到环时退回原始顺序，不阻塞执行）。
- **`getUpstreamData(targetNode, nodes, connections)`**：连线数据推断函数。`referenceNodeIds` 优先（显式声明），否则用入边推断。返回 `UpstreamData{imageHrefs,videoHrefs,audioHrefs,textContents,referenceNodeIds}`，供 `workflowGeneration.ts` 后续注入下游参考输入。
- **Script 连线规则**：`validateWorkflowConnection` 新增——Script 节点只能连向 image/video 节点；Script 节点只能接收 image/video/text 输入。
- **`group_nodes` op 处理**：给选中节点分配同一 `batchId` + `batchGroupSource`，复用现有 batch 折叠/展开 UI。
- **`ungroup_nodes` op 处理**：清除 `batchId`/`batchIndex`/`batchGroupSource`。
- **`execute_group` op 处理**：`topoSort` 排序后推入 `runRequests`，由 `applyOps` 依次调用 `onRunNode`。

### UI 集成

### ffmpeg 客户端（services/ffmpegClient.ts）
- **`getFFmpeg()`**：单例懒加载，首次调用 `FFmpegWASM.load()` 时根据 `isMultiThreadAvailable()` 决定加载单线程或多线程 core URL（`unpkg.com/@ffmpeg/core@0.12.6/dist/umd` 或 `@ffmpeg/core-mt@0.12.6`）。
- **`isFFmpegSupported()`**：检测 `SharedArrayBuffer` + `WebAssembly` 是否可用。
- **`isMultiThreadAvailable()`**：检测 `SharedArrayBuffer` + 多线程 core 可用性，决定是否启用 MT core。

### 视频工具服务（services/videoTools.ts）
- **`trimVideo(file, startSec, endSec)`**：用 ffmpeg `-ss` + `-to` + `-c copy` 流复制剪裁，不重编码，极快；返回 `Blob { type: video/mp4 }`。
- **`splitAudioVideo(file)`**：`-vn -c:a copy` 提取音频 + `-an -c:v copy` 提取纯视频；返回 `{ video: Blob, audio: Blob }`。
- **`mergeVideos(files)`**：先为每个输入生成 ts 片段（`-c copy -bsf:v h264_mp4toannexb`），再 concat demuxer 合并 + `-c copy` 输出 MP4；要求所有输入编码参数一致。
- **`getVideoDuration(file)`**：`-i` 解析 stdout 中 `Duration: hh:mm:ss.xx` 行。
- 所有命令统一通过 `ffmpeg.exec([...])` 调用；文件挂载/卸载通过 `ffmpeg.writeFile` / `ffmpeg.deleteFile`。

### 图像变换服务（services/imageTransform.ts）
- **`transformImage(src, action)`**：纯 Canvas 实现，`action` 支持 `rotate-90`/`rotate-180`/`rotate-270`/`flip-h`/`flip-v`；返回 PNG Blob。无 AI、无网络。

### 宫格切分服务（services/gridSplitter.ts）
- **`splitGrid(src, rows, cols)`**：纯 Canvas 按 `rows × cols` 切分源图，返回 `SplitGridResult[]`（含 `blob` + `row` + `col` + `index`）。无 AI、无网络。

### 音频工具服务（services/audioTools.ts — 新建）
- **`trimAudio(blob, startSec, endSec, name)`**：用 ffmpeg.wasm `-ss` + `-to` + `-c copy` 流复制截取音频片段，不重编码，极快；返回 `AudioTrimResult { blob, durationSec }`，输出格式延续源文件 MIME（默认 `audio/mpeg`）。

## Canvas 持久化 Bug 修复（Canvas ↔ Workflow 往返丢数据）

- **根因 #1 — `isSyncingRef` 异步时序**：`FlovartExcalidrawStage.tsx` 中 `pushFlovartToExcalidraw` 的 `finally` 块清 `isSyncingRef.current = false`，但 `updateScene` 是异步生效，Excalidraw 会在 flag 清除后再次触发 `onChange` → 重入。修复：不在 `finally` 中清，改在 `handleSceneChange` 中通过 `getSceneElements()` 的 ID 与 `lastExSnapshotRef.current` 比对确认同步完成后才清。
- **根因 #2 — `lastFlovartRef` 未更新**：`pushFlovartToExcalidraw` 在 booted 跳过 push 时也未更新 `lastFlovartRef.current`，导致下次 `handleSceneChange` diff 永远判定为"远端变更" → 死循环。修复：始终更新 ref，只在 booted 时跳过 `updateScene` push。
- **根因 #3 — `initialData` 覆盖 `updateScene`**：React 19 + StrictMode 下 `ExcalidrawInner` 的 `initialData` prop 在 `updateScene` 调用之后又被 React 重新应用，覆盖刚 push 的元素。修复：用 `setTimeout(0)` 延迟 push 到下一个事件循环，避开 React 重渲染窗口。
- **验证**：通过 Playwright 自动化在 Canvas 创建 2 个元素 → 切到 Workflow → 切回 Canvas，确认 2 items 完整保留、图层列表正确、Excalidraw scene 不被重置。
- **Debug trace 清理**：`[TRACE pushFlovart]` / `[TRACE sceneChange]` / `[TRACE guard]` / `[TRACE diff]` 四处 `console.warn` 已从 `FlovartExcalidrawStage.tsx` 移除。

## Canvas 图片不显示 Bug 修复（Excalidraw 0.18.1 图片渲染机制）

- **Excalidraw 0.18.1 图片渲染链路分析**（源码定位）：
  - 图片通过 **canvas 2D context.drawImage()** 渲染，不是 DOM `<img>` 标签；`.SVGLayer` 为空 `<svg></svg>` 是 0.18.x 正常行为。
  - 渲染链：`addFiles()` → `addMissingFiles()` → `addNewImagesToImageCache()` → `updateImageCache()` → `loadHTMLImageElement(dataURL)` → `imageCache.set(fileId, {image, mimeType})` → `scene.triggerUpdate()` → 重新渲染 → `drawElementOnCanvas` 从 `renderConfig.imageCache.get(fileId).image` 取 HTMLImageElement → `context.drawImage()`。
  - **关键发现**：`updateScene()` **不触发** `addNewImagesToImageCache()` — 必须通过 `addFiles()` 或 action 路径。`refresh()` 会强制 canvas 重绘但不重新加载 image cache。
- **根因 #1 — `baseFields` width/height 回退错误**：`utils/excalidrawAdapter.ts:117` 用 `overrides.width ?? 0`，但 Flovart ImageElement 的 `width` 字段在旧路径中可能未设置，应从 `boundsOf(el).width` 取真实宽高。修复：`baseFields` 改为 `overrides.width ?? bw ?? 0` / `overrides.height ?? bh ?? 0`（bw/bh 来自 `boundsOf(el)`）。此修复同时影响 `convertImage`/`convertVideo`/`convertShape`/`convertText`。
- **根因 #2 — `pushFlovartToExcalidraw` 调用顺序错误**：原顺序是 `registerImageFiles`（→ `addFiles` → `addNewImagesToImageCache`）在 `updateScene` 之前调用，但 `addNewImagesToImageCache` 扫描 scene 时图片元素还没进去 → imageCache 为空。修复：重排顺序为 **先 `updateScene`**（元素进 scene）→ **后 `registerImageFiles`**（`addFiles` → `addNewImagesToImageCache` 能找到元素）→ **`setTimeout(refresh, 120)`** 强制重绘。
- **验证**：通过 Playwright 自动化在 Canvas 创建红色 200×200 图片元素 → 像素采样确认 144 个红色像素 (255,0,0) 在正确坐标渲染，w:200 h:200 ✅。
