## 数据流改造：边升级为真实数据流（用户重新考虑配置节点后定方案）

**背景**：用户质疑"连接点有啥用"，坦白当前边是装饰性血缘标记（下游生图不读上游）。用户重新考虑配置节点，最终决策：不新增节点类型，把配置节点能力（上游读取+参数配置+批量生图）直接做进现有图片/视频节点。方案 B（轻量数据流）+ B1（边引用和 @引用合并去重）。

**关键发现**：
- `ElementGenerationState` 早已包含配置节点全部参数字段（provider/modelId/aspectRatio/durationSec/resolution/generateAudio/watermark），`CanvasElementBase.generationState` 让任何元素都能携带生图配置——"参数配置"能力已内嵌，无需新节点。
- `buildPromptPayload`（utils/promptPayload.ts）已预留 `graphInputs` 参数——图形输入接口，正是边的上游该走的路径。
- `handleBatchGenerate` 已存在批量生图，但结果图没带 generationState 也没建边——缺口。
- 真实差距只有一个：生图流程没读 edges 的上游。`handleGenerate` 只读 mentionedElementIds（手动@）和 selectedElementIds（选中）。

**核心语义**：选中节点 = 配置节点角色。选中节点的上游边 = 数据源。裸生图（PromptBar 没选中）无"当前节点"概念，只走 @引用。

**改动文件**：
- `hooks/useGeneration.ts`：
  - import 加 TextElement/CanvasEdge/getUpstreamIds
  - UseGenerationParams 加 `edges: CanvasEdge[]`
  - handleGenerate 开头计算 edgeUpstreamIds（getUpstreamIds 递归全上游）：图片/视频上游合并进 activeMentionedElementIds（B1 去重，buildCanvasGenerationState 的 seen Set 天然去重），文本上游拼进 prompt 末尾（`[上游文本: name]\n内容` 格式）
  - 6 处 buildCanvasGenerationState 的 referenceIds 加 `...edgeUpstreamIds`（keyframe/video/image-edit inpaint/image-edit 多图/image-mention/image-bare），让 resolvedReferences 记录完整血缘，App.tsx 已有 useEffect 自动建边
  - handleBatchGenerate：加 edgeUpstreamIds 注入 + 文本拼 prompt + batchReferenceIds（选中+@引用+上游）
  - batchResults state 加 referenceIds 字段
  - handleSelectBatchResult / handleSelectAllBatchResults：结果图带 generationState（含 referenceIds），让自动建边 useEffect 接管
- `App.tsx`：useGeneration 调用处加 `edges: activeBoard?.edges || []`

**数据流闭环**：拖边建图 → 选中下游节点生图 → 自动读上游内容（图片作参考图，文本拼 prompt）→ 结果图带 generationState → useEffect 自动建边到上游 → 血缘图完整。边从装饰性血缘升级为真实数据流。

**待验证**：递归建边是否让图过乱（深链路时结果图连到所有间接上游）；文本上游拼 prompt 的格式是否影响生图效果。
